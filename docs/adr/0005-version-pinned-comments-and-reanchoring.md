# ADR-0005: コメントの版固定と「退避して保持」な再アンカー

- Status: Accepted
- Date: 2026-06-28

## Context

インラインコメント（[ADR-0004](./0004-inline-anchored-comments.md)）は本文のテキスト範囲に固定される。
ところがこのツールの本質は「AI に直させて HTML を差し替える」反復であり、**本文は頻繁に変わる**。
追記・整形・書き換えが起きると、アンカーは位置がずれ、最悪は対象テキストごと消える。

このとき素朴な実装は「コメントが見当違いの箇所を指す」事故を起こす。これはレビューの信頼を根本から壊すため、
**絶対に許容できない**。「ずれるなら、黙ってずれるな」をどう保証するかが論点。
あわせて、そもそもコメントを「どの版に対するものか」固定できなければ追従の議論が始まらない。

## Considered Options

アンカーの版への結びつけ方：

1. **版に紐付けない**（コメントは files に直結。最新本文に対してだけアンカー）
2. **版に固定（version-pinned）**（コメントはコメント時点の版に紐づく）

新版が来たときの未解決コメントの扱い：

- **A: 持ち越さない**（付けた版だけに表示。新版へは引き継がない）
- **B: 厳密一致のみ持ち越し**（引用が完全一致する箇所だけ自動追従、変われば即クローズ）
- **C: 退避して保持**（引用＋文脈で再マッチ。当たれば追従、外れたら orphaned として退避しスレッドは保持）

## Decision

**「版に固定（選択肢2）」＋「退避して保持（C）」を採用。**

- 各コメントはコメント時点の `fileVersions` 行に固定する。R2 blob はイミュータブルなので、
  **その版を見ている限りアンカーは絶対にずれない**。
- 新版アップロード時、未解決コメントを**引用テキスト＋前後文脈で新版から探し直す**：
  - 見つかった → 位置がずれても新版へ再アンカーして**追従**。
  - 見つからない → **「変更された箇所」として退避（orphaned）**。スレッド本文と旧引用は保持し、
    レビュアーが手動で現在地へ再リンク／解決できる。
- この版固定を成立させるため、**バージョンの薄い土台**（アップロードごとに `fileVersions` を1行作り、
  `files.currentVersionId` が指す。V1 既存資料は現 blob を「版1」にバックフィル）を、
  コメント機能と**同じフェーズで同時に導入**する。

## Consequences

- **Positive:**
  - 「黙ってズレて誤った箇所を指す」事故が構造的に起きない。追従できなければ正直に退避する。
  - 反復（差し替え）をしても、付いていた指摘が可能な限り新版へ引き継がれ、会話が分断されない。
  - バージョンの土台が先に入るため、Phase 3 の本格バージョン管理 UI はこの上に乗せるだけで済む。
- **Negative / トレードオフ:**
  - 再マッチ（ファジー検索）と orphaned 状態の UI という実装コストが乗る。
  - 大幅な書き換えでは追従率が下がり、orphaned が増える。→ それでも「消える」より「退避で残す」を選ぶ。
  - コメント機能だけのつもりが、版の土台導入（スキーマ運用・バックフィル）まで巻き込む。
    → だが版固定の前提なので不可分。ここで払うのが最も安い。

## なぜ他案を退けたか

- **版に紐付けない（選択肢1）:** 最新本文だけにアンカーするため、古い版を見ても指摘が再現できず、
  追従の成否も判定できない。反復ツールとしての履歴性を失う。
- **持ち越さない（A）:** 再アンカー不要で最もシンプルだが、版を上げるたび会話が分断され、
  「前の版で指摘したのに新版で消えた」が頻発する。反復ループの価値を削ぐ。
- **厳密一致のみ（B）:** orphaned 一覧を持たずに済む代わり、わずかな整形でも指摘が即クローズされ、
  レビューの取りこぼしになる。「取りこぼすより、退避して人に判断させる」を優先した。
