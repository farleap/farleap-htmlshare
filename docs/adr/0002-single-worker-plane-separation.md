# ADR-0002: 単一 Worker・パス分離による面の同居

- Status: Accepted
- Date: 2026-06-28（V1 で確定した決定を遡及的に記録）

## Context

このツールには性質の異なる2つの面がある：

- **App 面** — 認証されたユーザーの操作（アップロード・一覧・詳細・共有・削除）。身元が確定した信頼領域。
- **Content 面** — 信頼しない生 HTML の配信。隔離が最優先。

素朴には「2つを別オリジン（別ドメイン/別 Worker）に分ければ、eTLD+1 レベルで最も強く分離できる」と考えられる。
一方、本物の隔離は [ADR-0001](./0001-opaque-origin-iframe-isolation.md) の sandbox iframe（不透明オリジン）が既に担っている。
そこで「面分離のために独自ドメインや複数デプロイを運用する価値があるか」が論点になった。

## Considered Options

1. **2つの独立した Worker / 独自ドメインで物理分離**（App と Content を別ホストに）
2. **単一 Worker に同居させ、パスで App / Content を出し分ける**

## Decision

**選択肢 2 を採用。** 1つの Worker（単一 `workers.dev` ホスト）に両面を同居させ、
リクエストの**パス**で App 面と Content 面を振り分ける。Content 面のパスだけを生 HTML 配信に割り当てる。
ホスト全体に Cloudflare Access を掛け、ログイン済みユーザーの iframe は Cookie で Content 面を通過しつつ、
**署名トークンでも保護**する（多層）。

ただし**二ホスト構成への対応余地は残す**：振り分けロジックは「App ホスト ≠ Content ホスト」のときに
ホスト名でも分離できるよう書いてあり、より強い eTLD+1 分離が必要になれば Content 面を独自ドメインへ移せる。

## Consequences

- **Positive:**
  - 独自ドメイン・DNS・追加デプロイが不要。運用が単純で、`workers.dev` だけで完結。
  - 隔離は sandbox iframe が担保するため、ホストを分けなくても主防御は揺るがない。
  - ローカル dev では `127.0.0.1` と `localhost` を App/Content に割り当てることで、
    単一サーバのまま本番のクロスオリジン分離を再現・テストできる。
- **Negative / トレードオフ:**
  - 同一ホストゆえ、eTLD+1 レベルの分離は得られない（Cookie スコープ等は共有される）。
    → 主防御は不透明オリジンであり、ここは「より強くもできる」余地として許容。
  - パス振り分けの条件分岐に、面の境界というセキュリティ上重要なロジックが乗る。
    → テストで境界（`/p/*` だけが Content）を固定する。
- **撤回条件:** 社外共有（[DESIGN.md §7](../DESIGN.md)）など、より強い分離が要件化したら、
  二ホスト構成へ移行する（コードは対応済み）。その際この ADR を `Superseded` にせず、新 ADR で上書きする。
