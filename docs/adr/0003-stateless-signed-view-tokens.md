# ADR-0003: ステートレス HMAC 署名トークンによる配信認可

- Status: Accepted
- Date: 2026-06-28（V1 で確定した決定を遡及的に記録）

## Context

Content 面（生 HTML の配信）には認可が要る。推測可能な URL だけで生 HTML を取得できてしまうと、
隔離以前に「中身が漏れる」。一方、配信されるドキュメントは不透明オリジンに置かれ Cookie を持てない
（[ADR-0001](./0001-opaque-origin-iframe-isolation.md)）ため、Cookie ベースのセッションには素直に乗れない。

「誰が・どのファイルを・いつまで見てよいか」を、Cookie に依存せず、かつ状態を増やさずに表現したい。

## Considered Options

1. **サーバ側セッション**（セッション ID を発行し、サーバ/ KV に状態を保持して照合）
2. **Cloudflare Access の Cookie のみ**で配信を保護
3. **HMAC 署名したステートレスな短命トークン**を URL に載せる

## Decision

**選択肢 3 を採用。** `{ fileId, email, exp }` を `TOKEN_SECRET` で HMAC-SHA256 署名した
短命トークンを発行し、Content 面はこれを検証して配信する。署名と有効期限だけで真正性を判定でき、
**サーバ側に状態を持たない**。トークンは数分で失効する。

実運用では Access Cookie（ホスト全体の認証）と**併用**し、多層防御とする。

## Consequences

- **Positive:**
  - サーバ側セッションストアが不要。Workers のステートレス性と相性が良くスケールする。
  - 短命なので、URL が漏れても時間経過で自然に無効化される。
  - 「どのファイルを」までトークンに焼き込めるので、トークンの使い回しで別ファイルを取ることはできない。
- **Negative / トレードオフ:**
  - 発行済みトークンを有効期限前に**個別失効**はできない（ステートレスの代償）。
    → 有効期限を短く保つことで露出窓を最小化。長期の共有は別レイヤ（共有リンク）で扱う。
  - `TOKEN_SECRET` の漏洩は全トークン偽造に直結する。→ secret 管理を厳格にする。
- **不変条件化:** 「Content 面は有効な署名トークンが無ければ 403」を回帰テストで固定する。
