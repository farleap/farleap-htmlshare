# ADR-0001: 不透明オリジン iframe によるプレビュー隔離

- Status: Accepted
- Date: 2026-06-28（V1 で確定した決定を遡及的に記録）

## Context

このツールの中心は「他人（多くは AI）が作った HTML を、社内の人がブラウザで開く」こと。
だが他人の HTML を開く＝**任意の JavaScript を閲覧者のブラウザで実行する**ことであり、
何もしなければ閲覧者の Cookie・セッション・ページ内データの窃取、なりすまし、フィッシングが成立しうる。

一方で、HTML の価値（インタラクティブな試算ツール、動くダッシュボード等）を保つには、
**JS を完全に殺すこともできない**。「JS は動かしたいが、害は一切与えさせない」を同時に満たす必要がある。

## Considered Options

1. **同一オリジンで配信し、サニタイズで無害化**（DOMPurify 等でスクリプト/イベントを除去）
2. **CSP のみで縛る**（`script-src` を厳しくして実行を制限）
3. **不透明オリジンの sandbox iframe に隔離**（`allow-scripts` は付けるが `allow-same-origin` を付けない）

## Decision

**選択肢 3 を採用。** プレビューは別経路（Content 面）から配信し、App 側の詳細画面では
`sandbox="allow-scripts ..."` の iframe に埋め込む。**`allow-same-origin` は決して付けない。**

これにより iframe 内のドキュメントは「どのオリジンとも同一でない不透明オリジン」に置かれ、
JS は動くが、`document.cookie`・親フレームの DOM・`localStorage`・他タブ等、
オリジンに紐づくあらゆるものへ到達できなくなる。**隔離をブラウザの保証に乗せる**のが核心。

## Consequences

- **Positive:**
  - 「安全かどうか」を自前のサニタイズ実装の完全性に賭けずに済む。ブラウザのオリジン境界に依拠できる。
  - HTML 内の JS をそのまま動かせる（プロダクト価値を損なわない）。
  - これがツール全体の**主防御**となり、他の対策（トークン・CSP）はこれを補強する位置づけになる。
- **Negative / トレードオフ:**
  - 不透明オリジンゆえ、親（App）から iframe の中を**直接読めない/操作できない**。
    → 後にコメント機能でインライン選択を実現する際、「注入スクリプト＋ postMessage」という
      間接経路を取らざるを得ない（[ADR-0004](./0004-inline-anchored-comments.md) の前提制約）。
  - iframe 内コンテンツも Cookie を持てないため、配信の認可は別手段（署名トークン）が必要
    （[ADR-0003](./0003-stateless-signed-view-tokens.md)）。
- **不変条件化:** 「`allow-same-origin` を付けない」は回帰テスト（unit / E2E）で固定し、
  これを破る変更は設計違反とみなす。

## なぜサニタイズ案（選択肢1）を退けたか

サニタイズは「危険なものを列挙して除去する」ブラックリスト的発想で、回避手法とのいたちごっこになりやすく、
「絶対に安全」を保証しきれない。隔離（選択肢3）は「そもそも到達させない」ホワイト境界であり、保証の質が違う。
