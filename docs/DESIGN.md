# farleap-htmlshare — 設計ドキュメント（完成形）

このドキュメントは **farleap-htmlshare の「あるべき姿（North Star）」** を一気通貫で言語化したもの。
現状の実装（V1）、追加予定のコメント/レビュー・バージョン管理、そしてその先の将来像までを、
*何を実現するか（What）* と *なぜそう設計するか（Why）* のレベルで記述する。

実装の詳細（How: 具体的なヘッダ値・関数・プロトコルの細部）は各実装フェーズで詰めるため、
ここでは深入りしない。**設計判断と不変条件が変わらない限り、このドキュメントが唯一の正典**とする。

関連ドキュメント:

- **個々の重い設計判断の経緯** → [`docs/adr/`](./adr/)（Architecture Decision Records）
- **本番デプロイ構成・再現手順** → [`docs/deploy.md`](./deploy.md)

各機能には実装状況を明記する：

- ✅ **実装済み** — 本番稼働中（V1）
- 📝 **設計済み・未実装** — 次フェーズ。本ドキュメントで仕様を確定
- 🔮 **将来** — データモデル上は布石を打ってあるが、仕様は未確定

## 1. このツールは何か

**社内向けの「AI生成HTMLを、安全に・見せて・論じて・磨く」ための共有基盤。**

AI（Claude など）が吐き出す成果物は、いまや自己完結した1枚の HTML であることが多い。
レポート、ダッシュボードのモック、インタラクティブな試算ツール、LP のドラフト——
これらは「ファイルで送る」には惜しく、「どこかにデプロイする」には重い。
そして **他人が作った HTML を開く＝任意コードを自分のブラウザで実行する** という、無視できない危険を伴う。

farleap-htmlshare は、この隙間を埋める。アップロードした瞬間に社内の誰とでも共有でき、
**危険な HTML でも閲覧者の環境を一切侵せない**よう隔離し、不要になれば自動で消える。

### 完成形の一文

> **AI が生成した HTML を、アップロードした瞬間に社内へ安全に共有でき、文中を選んでコメントでレビューし合い、修正版を重ねて履歴を残せる。どんな HTML を踏んでも、閲覧者の Cookie・親ページ・他タブには指一本触れさせない。**

## 2. North Star — 3つの体験

完成形は、ひとつの資料に対する **3つの動詞** で説明できる。

| 体験 | 動詞 | 状態 | 何を可能にするか |
|---|---|---|---|
| **Share（共有）** | 見せる | ✅ V1 稼働中 | HTML を上げ、社内の誰でも開けるリンクを即発行。90日で自動消去。 |
| **Review（レビュー）** | 論じる | 📝 設計済み | プレビュー上で文中を選び、インラインでコメント・スレッド・解決。 |
| **Iterate（反復）** | 磨く | 📝 設計済み | 修正版を重ねて差し替え、履歴を残す。コメントは版を跨いで追従する。 |

この3つが噛み合うと、**「AI に作らせる → 社内で見せる → コメントで指摘する → AI に直させて差し替える → 指摘が新版に追従する」** という反復ループが、ひとつのツールの中で完結する。これがこのプロダクトの到達点。

```
        ┌──────────────────────────────────────────────┐
        │                                              │
        ▼                                              │
   AI が HTML 生成 ──▶ アップロード ──▶ 社内で共有・閲覧      │
                                          │           │
                                          ▼           │
                                   文中を選んでコメント      │
                                   （レビュー）            │
                                          │           │
                                          ▼           │
                                   指摘を AI に戻す ────────┘
                                   修正版を差し替え
                                   （コメントは新版へ追従）
```

## 3. 設計原則（壊してはいけないもの）

機能は足していくが、**以下は全フェーズを通じて不変**とする。新機能がこの原則と衝突する場合、機能ではなく設計を変える。

### セキュリティ不変条件

このツールの存在意義は「危険かもしれない HTML を安全に扱う」ことなので、セキュリティは機能ではなく**土台**である。

1. **不透明オリジン隔離が主防御。** プレビューは別オリジンの iframe に `sandbox="allow-scripts ..."` で埋め込み、**`allow-same-origin` を絶対に付けない**。これにより HTML 内の JS は動くが、不透明オリジンに閉じ込められ、親ページの DOM・Cookie・localStorage・他タブへ一切到達できない。（→ [ADR-0001](./adr/0001-opaque-origin-iframe-isolation.md)）
2. **コンテンツ配信は署名トークン必須。** 生 HTML を返す経路（Content 面）は、HMAC 署名された短命トークンが無ければ 403。推測可能な URL だけでは中身を取れない。（→ [ADR-0003](./adr/0003-stateless-signed-view-tokens.md)）
3. **配信レスポンスは厳格ヘッダ固定。** `Content-Security-Policy`（埋め込み元を App に限定）+ `X-Content-Type-Options: nosniff`、かつ **`Set-Cookie` を絶対に付けない**。
4. **アップロードはサーバ側で検証。** MIME・拡張子・サイズ上限はサーバが判定し、クライアント申告値を信用しない。HTML のみ・上限あり。
5. **識別子はすべて不透明。** ファイル ID・トークンはランダム生成。連番や内部パスをクライアントへ晒さない。所有・参照の権限は必ず DB 引きで検証する。

> これらは回帰テスト（unit / integration / E2E）で守る。**テストが落ちる変更は、設計違反**とみなす。

### プロダクト設計原則

- **単一 Worker で面を分離。** App 面（認証された操作）と Content 面（生 HTML 配信）を1つの Worker に同居させ、パスで出し分ける。運用は単純なまま、ブラウザから見れば別オリジンの隔離が効く。（→ [ADR-0002](./adr/0002-single-worker-plane-separation.md)）
- **ステートレスなトークン。** 配信トークンは署名のみで検証でき、サーバ側セッションを持たない。短命（数分）で、漏れても時間で死ぬ。
- **ソフトデリート + 定期 purge。** 削除は即時に不可視化（論理削除）し、実体の消去は日次ジョブに委ねる。誤操作からの猶予と、確実な自動消去（90日）を両立する。
- **漸進的開示。** 普段の閲覧は最小限の表面積で行い、レビューなど能動的な操作のときだけ機能（とコード注入）を有効化する。常時有効にしない。
- **イミュータブルな実体、ミュータブルなメタ。** R2 上の HTML blob は一度書いたら書き換えない（差し替えは新 blob）。これがバージョン管理とコメント追従の前提を成す。

## 4. アーキテクチャ概観

### 面（plane）の分離

| 面 | 役割 | 認証 | 性質 |
|---|---|---|---|
| **App 面** | アップロード・一覧・詳細・共有リンク・削除・（将来）コメント/版操作 | 社内 Google 認証（Cloudflare Access、許可ドメイン限定） | 信頼された操作。ユーザーの身元が確定している。 |
| **Content 面** | アップロードされた生 HTML の配信 | 署名トークン（＋ログイン済み Cookie で Access 通過） | 信頼しないコンテンツ。隔離が最優先。 |

本番は両面を単一ホストに同居させ、**パスで分離**する。Content 面のパスだけを生 HTML 配信に割り当て、残りを App 面とする。
（より強い分離が要れば、Content 面を独自ドメインへ移して eTLD+1 レベルで切り離せる設計余地を残している。→ [ADR-0002](./adr/0002-single-worker-plane-separation.md)）

### ストレージ

- **R2** — HTML blob（イミュータブルな実体）。
- **D1** — メタデータ（ファイル・共有リンク・版・コメント・権限）。Drizzle で型付け。

### データモデル（完成形の全体像）

完成形で扱う実体は5つ。**現状すでにスキーマ定義は存在し**、V1 では一部だけを使っている。
追加機能はテーブルを新設するのではなく、**既に打ってある布石を起動する**形で進む。

| 実体 | 役割 | 利用フェーズ |
|---|---|---|
| **files** | 資料1件のメタ（所有者・タイトル・現在版・可視性・期限・pin・論理削除） | ✅ V1 |
| **shareLinks** | 共有リンク（トークン・スコープ・失効） | ✅ V1（`scope` は将来拡張の布石） |
| **fileVersions** | 1ファイルの各版（版ごとの blob・作成者・メモ） | 📝 バージョン管理（V1 では未使用） |
| **comments** | コメント（本文・著者・解決状態・親子＝スレッド・**テキストアンカー**） | 📝 コメント/レビュー |
| **permissions** | ファイル単位の人×ロール権限 | 🔮 将来 |

```
files (資料)
  ├─< fileVersions (版)        ← 📝 各アップロードが1版に
  │     └─< comments (版に固定) ← 📝 コメントは「付けた版」に紐づく
  ├─< shareLinks (共有リンク)   ← ✅ V1
  └─< permissions (権限)       ← 🔮 将来
```

## 5. 主要な設計判断（ADR 索引）

「なぜこの設計にしたか／なぜ別案を退けたか」は、後から最も蒸し返されやすく、最も価値が高い。
ひとつひとつの重い決定は **ADR（Architecture Decision Record）** に経緯ごと凍結し、ここではその一覧と要約だけを示す。
詳細・却下理由・撤回条件は各 ADR を参照。

| ADR | 決定 | 主な代替案 | 退けた理由（要約） |
|---|---|---|---|
| [0001](./adr/0001-opaque-origin-iframe-isolation.md) | 不透明オリジン iframe で隔離（`allow-same-origin` を付けない） | 同一オリジン＋サニタイズ（DOMPurify 等）／CSP のみ | サニタイズはいたちごっこで「安全」を保証しきれない。隔離はブラウザの保証に乗れる。 |
| [0002](./adr/0002-single-worker-plane-separation.md) | App 面と Content 面を単一 Worker にパス分離で同居 | 2つの独立 Worker／別ドメイン分離 | 独自ドメイン・複数デプロイの運用コストに見合わない。隔離は sandbox が担うのでホスト分離は必須でない。 |
| [0003](./adr/0003-stateless-signed-view-tokens.md) | 配信は HMAC 署名のステートレス短命トークン | サーバ側セッション／Access Cookie のみ | セッションは状態管理が増える。Cookie のみだと URL 推測で生 HTML に到達しうる。 |
| [0004](./adr/0004-inline-anchored-comments.md) | コメントはインライン（範囲アンカー）方式 | 文書全体への単一スレッド | 「どこの話か」を本文に固定できないと、AI 生成物の具体的な指摘に使えない。 |
| [0005](./adr/0005-version-pinned-comments-and-reanchoring.md) | コメントを版に固定し、新版へは「退避して保持」で再アンカー（＋版の薄い土台を同時導入） | 版に紐付けない／版固定で持ち越さない／厳密一致のみ持ち越し | 黙ってズレる事故は許容不可。持ち越さないと反復のたびに会話が分断。土台が無いと版固定自体が成立しない。 |

## 6. 機能仕様

### 6.1 ✅ Share（共有）— V1・実装済み

現状の本番。完成形における「土台の一面」。

- **アップロード** — 自己完結 HTML を1ファイル、ドラッグ＆ドロップで即アップロード。サーバ側で HTML 検証（MIME・拡張子・サイズ上限）。
- **ダッシュボード** — 自分と社内の資料を一覧。タイトル・所有者・残り保存日数を表示。
- **プレビュー** — 詳細画面で、隔離 iframe に実物をレンダリング。HTML 内の JS は動くが親環境には到達不可。
- **共有リンク** — 社内の誰でも開ける不透明リンクを発行。Access 認証は通すので「社内限定で誰でも」。
- **削除 / pin** — 所有者は論理削除でき、pin した資料は自動削除の対象外。
- **自動削除** — 日次ジョブが、90日経過・非 pin・未削除の資料を実体ごと purge。

**設計上の含意:** 「アップロード = 1つの blob = 1件の files 行」という素朴な対応。バージョン管理はこれを「1件の files が複数の版を持つ」へ自然拡張する（§6.3）。

---

### 6.2 📝 Review（レビュー）— コメント機能・設計済み

**目的:** AI 生成物に対して、社内のレビュアーが**文書の該当箇所を指しながら**フィードバックを残せるようにする。「3段落目のこの数字おかしくない？」を、口頭やチャットの曖昧な参照ではなく、**本文に固定されたコメント**として残す。

#### 体験

- プレビュー詳細画面に **「レビューモード」トグル** を置く。普段は今まで通りの純粋な閲覧（余計なものは何も注入しない）。レビューしたいときだけ ON にする。
- レビューモード ON で、画面が **プレビュー（左）＋コメントパネル（右）** の2ペインになる。
- プレビュー内の**テキストを範囲選択**すると吹き出しが出て、その箇所に紐づくコメントを書ける（Google Docs のインラインコメントのイメージ）。
- 既存コメントは、対応する本文箇所が**ハイライト**され、クリックでパネル側のスレッドに飛ぶ。
- コメントは**スレッド化**（返信）でき、**解決（resolved）**で畳める。
- パネル上部に**未解決件数**を常時表示。

```
レビューモード ON:
┌─ プレビュー ───────────┬─ コメント (未解決 2) ──┐
│  見出し                │ ▸ 田中: この数値の根拠は？│
│  ┃本文のこの範囲┃ ←選択 │   └ 菊地: v2で直す ✓     │
│  次の段落…             │ ▸ 佐藤: 見出し弱い        │
│                        │ ─────────────────       │
│                        │ [選択箇所にコメント…]      │
└────────────────────────┴──────────────────────────┘
```

#### 設計判断

- **インライン（範囲アンカー）方式を採用。** 文書全体への単一スレッドではなく、本文の特定範囲にコメントを固定する。`comments` テーブルのアンカー列（引用テキスト＋前後文脈＋位置）がこれを支える。（→ [ADR-0004](./adr/0004-inline-anchored-comments.md)）
- **隔離は壊さない。** インライン選択を成立させるには、Content 面が配信する HTML に「選択を検知して親へ受け渡す／既存コメントをハイライトする」ための**表示用スクリプトを注入**する。ただし:
  - iframe は従来通り**不透明オリジンのまま**（`allow-same-origin` は付けない）。注入されたスクリプトも親の Cookie・DOM には到達できない。
  - 親 ⇄ iframe の通信は **postMessage** で行い、**双方が相手のオリジンを厳密に検証**する。
  - 注入は**レビューモード ON のときだけ**。通常閲覧では一切注入しない（漸進的開示）。
  - 本文の DOM は書き換えず、ハイライトは非破壊の手段で重ねる。
  - → **セキュリティ不変条件 1〜5 はすべて維持される。**
- **コメントは「付けた版」に固定する（version-pinned）。** 各コメントは、コメント時点のバージョンに紐づく。R2 の blob はイミュータブルなので、**その版を見ている限りアンカーは絶対にずれない**。（→ [ADR-0005](./adr/0005-version-pinned-comments-and-reanchoring.md)）
- **新版での追従は「退避して保持」。** 修正版がアップロードされたら、未解決コメントを**引用テキスト＋前後文脈で新版から探し直す**：
  - 見つかった → 位置がずれていても**新版に再アンカーして追従**。
  - 見つからない（AI がその箇所を書き換えた）→ **「変更された箇所」として退避（orphaned）**。スレッド本文と元の引用は保持し、レビュアーが手動で現在地へ貼り直す／解決できる。
  - **黙ってズレて誤った箇所を指すことは起こさない。** 追従できたら追従、できなければ正直に「ここは変わった」と示す。

```
新版アップロード後:
┌─ プレビュー ──────┬─ コメント ─────────────┐
│ 本文…             │ ▸ 田中(追従) ✓          │
│ ┃追従できた範囲┃   │ ────────────────       │
│                   │ ⚠ 変更された箇所 1        │
│                   │ ▸ 佐藤: ここ直して        │
│                   │   「（旧引用文を表示）」    │
│                   │   [現在地に再リンク]       │
└───────────────────┴──────────────────────────┘
```

#### 検討した代替案

| 案 | 内容 | 退けた理由 |
|---|---|---|
| 文書全体スレッド | アンカー無しで「資料1件への1スレッド」 | 実装は最も軽いが、「どこの話か」が本文に固定されず、AI 生成物の具体的指摘という主目的を満たさない。（[ADR-0004](./adr/0004-inline-anchored-comments.md)） |
| 版固定で持ち越さない | コメントは付けた版だけに表示、新版へは一切引き継がない | 再アンカー不要で最もシンプルだが、反復のたびに会話が分断され「前の版で指摘したのに」が頻発。（[ADR-0005](./adr/0005-version-pinned-comments-and-reanchoring.md)） |
| 厳密一致のみ持ち越し | 引用が完全一致する箇所だけ自動追従、少しでも変われば即クローズ | orphaned 一覧を持たずに済むが、わずかな整形でも指摘が消え、レビューの取りこぼしになる。 |

#### 権限・範囲

- **コメントできる人:** 社内認証済みのユーザーは誰でも（所有者に限らない）。レビューは協働作業なので、閲覧できる＝コメントできる、を既定とする。
- **編集・削除:** 自分が書いたコメントは編集・削除できる。
- **解決:** スレッドの解決は、コメント参加者と資料の所有者ができる。

#### 非ゴール（このフェーズではやらない）

- **外部（メール / Slack）通知。** まずはアプリ内の未解決件数表示に留める。通知は将来フェーズ（§7）。
- **リアルタイム共同編集 / 同時カーソル。** これは「HTML を丸ごと差し替える」反復モデルであって、ライブ共同編集ではない。
- **HTML 本文そのものの編集。** コメントは本文に重ねるだけ。本文の修正は「AI に直させて新版を差し替える」で行う（§6.3）。

---

### 6.3 📝 Iterate（反復）— バージョン管理・設計済み

**目的:** AI との「指摘 → 修正 → 差し替え」の反復を、**履歴として残しながら**回せるようにする。資料の URL（ID）は変えずに中身を更新でき、過去版にいつでも戻れる。

#### 体験

- 既存資料に対して**「新しい版をアップロード」**できる。資料の ID・共有リンクはそのまま、中身だけが最新版に変わる。
- 詳細画面に**版セレクタ**を置き、過去の版を閲覧できる。各版には作成者・日時・任意のメモ（「フィードバック反映」等）が付く。
- 新版を上げると、前版に付いていた未解決コメントが**新版へ追従を試みる**（§6.2 の「退避して保持」）。

#### 設計判断

- **薄い土台はコメント機能と同時に入れる。** コメントが「版に固定」される以上、版の概念はコメント機能の前提である。よって **「アップロードのたびに `fileVersions` を1行作り、files がその版を現在版として指す」** という最小の版管理を、コメント機能と同じフェーズで導入する。V1 の既存資料には、現在の blob を「版1」としてバックフィルする。（→ [ADR-0005](./adr/0005-version-pinned-comments-and-reanchoring.md)）
- **本格的なバージョン管理 UI（差し替えフロー・版セレクタ・履歴・差分表示）は、その上に乗せる。** 土台が先、UI が後。
- **差分（diff）表示は将来の上積み。** まずは「版を切り替えて見られる」「コメントが追従する」までを完成形の必須とし、版間の差分ハイライトはその先の改善とする。

#### 検討した代替案

| 案 | 内容 | 退けた理由 |
|---|---|---|
| バージョン管理を完全に後回し | コメントを先に出し、版概念は後で足す | コメントが版に固定できず、再アップロード時に追従ロジックを後付けする羽目になる。土台だけは先に敷くのが正しい。 |
| 上書きアップロード（履歴なし） | 新版で blob を上書きし、過去版を残さない | 反復の履歴・ロールバックが失われる。「磨く」体験の核を捨てることになる。 |
| マージ／分岐モデル | 版が枝分かれし、統合できる | このツールの反復は一直線（AI に直させる→差し替え）。分岐の複雑さは要らない。 |

#### 非ゴール

- **版ごとの細かい権限分岐**（この版は誰々だけ等）。版は資料の権限を継承する。
- **マージ / 分岐。** 版は一直線の履歴。枝分かれはしない。

## 7. 🔮 さらに先 — 将来像

データモデルに布石はあるが、仕様は未確定。完成形の「外周」として方向性だけ記す。

- **ファイル単位の権限（`permissions`）** — 現状の「社内なら誰でも閲覧/コメント可」を超えて、資料ごとに人とロール（閲覧者／コメント可／編集可）を指定する。機微な資料の限定共有に。
- **社外共有（`shareLinks.scope`）** — 現状の共有は社内 Google 認証の内側に限られる。`scope` を社外向けに拡張し、Access の外の相手にも**署名トークンだけで**安全に見せる導線。セキュリティ不変条件の再設計を要するため、最も慎重に進める領域。
- **可視性制御（`files.orgVisibility`）** — 社内でも「組織全体に見せる／限定する」を出し分ける。
- **通知** — コメント・新版を、所有者やスレッド参加者へメール / Slack で通知。レビューの非同期性を強化する。
- **一覧 UX の強化** — サムネイル、検索・フィルタ、所有者別ビュー。

## 付録: 用語

- **App 面 / Content 面** — 認証された操作の面と、生 HTML を配信する面。単一 Worker・パス分離。
- **不透明オリジン（opaque origin）** — `allow-same-origin` なしの sandbox iframe が置かれる、どのオリジンとも同一でない隔離空間。このツールの主防御。
- **署名トークン** — Content 面アクセスに必須の、HMAC 署名された短命トークン。
- **version-pinned コメント** — コメント時点の版に固定されたコメント。その版では位置がずれない。
- **退避（orphaned）** — 新版で引用箇所が見つからず、本文に再固定できなくなったコメント。スレッドと旧引用は保持され、手動で再リンクできる。
